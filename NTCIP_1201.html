1 General
NTCIP_1201-2 Text

1.1 Scope
NTCIP_1201-3 Text

The messaging between a Transportation Management Center and its field devices is accomplished by using the NTCIP Application Layer services to convey requests to access or modify values stored in a given device; the parameters used to store these values are referred to as objects. Objects are instances of object types; some object types have singular instances within a device, while others can occur in conceptual tables and have multiple instances. This document identifies and defines object types that may be supported by multiple device types (e.g., actuated signal controllers and dynamic message signs). In the NTCIP family of standards, object types for a given device type are grouped in a device-type-specific data dictionary standard.

NTCIP_1201-90 Text

Sections 2 and 3 define objects that represent the current preferred design. Sections 4 through 16 define objects that are deprecated and represent designs from previous versions of NTCIP. The object types defined in this document are expected to be used by different device types, such as actuated signal controllers (ASC), variable message signs, and ramp meter controllers. The object types are defined using the OJBECT-TYPE macro defined in RFC 2578 per the rules defined in NTCIP 8004.

NTCIP_1201-214 Text

The MODULE-COMPLIANCE statements contained within each MIB duplicates and extends requirements that may be contained within the PRL of that standard. A device-specific module (e.g., the module for NTCIP 1202) will typically contain one MODULE-COMPLIANCE statement for each revision of the module. Each version will import object groups as needed and identify any appropriate constraints for implementation within that device type. In case of any conflict between the PRL and a referenced MODULE-COMPLIANCE statement, the PRL take precedence. To conform to a MIB defined within this document, implementations shall:

* support the defined MAX-ACCESS of each supported object, unless the associated MIB compliance table indicates a different minimum access for the object type;
NTCIP_1201-198 Text

* support at least the defined minimum access for each object type when one is indicated in the associated MIB compliance table,
NTCIP_1201-94 Text

* support the full range of values defined by the SYNTAX for each supported object, unless the associated MIB compliance table indicates a refined SYNTAX for the object type;
NTCIP_1201-95 Text

2.2 Current Situation and Problem Statement [informative]
NTCIP_1201-159 Text

Transportation system managers use ITS field devices in a variety of ways to improve transportation system operations. To perform their jobs, transportation system managers need to obtain data from the field (e.g., via sensors, cameras), implement traffic control policies (e.g., via gates, signals), and convey information to the traveling public (e.g., via message signs, RSUs). ITS field devices can permanent, temporary, or transportable and can use a wide variety of communications media. This standard is intended to support all of these environments.
NTCIP_1201-160 Text

Despite the varied nature of these devices, users expect every device to follow some basic principles in data exchange and there are advantages in standardizing some features (e.g., event logging) across device types. This standard focuses on the common user needs, requirements, and design elements that are common across many device types.
NTCIP_1201-161 Text

2.2.1 Backward Compatibility and Support of Prior Versions of NTCIP Data
NTCIP_1201-162 Text

The enhancement of certain functions over time have caused certain objects to be deprecated and/or replaced. Most updates to NTCIP standards are mostly backwards compatible, meaning that:
NTCIP_1201-163 List Level 1

Most objects remain current and
NTCIP_1201-164 List Level 1

Objects deprecated by the new version can still be supported and exchanged by a field device claiming conformance to the new version (if the device supports the object and recognizing the limitations and issues that caused the object to be deprecated).
NTCIP_1201-165 Text

NTCIP 1201 v04 does not follow this rule because it is based on a different protocol (SNMPv3 rather than SNMPv1). As a result, the objects and dialogs defined by this standard are not directly interoperable with prior NTCIP implementations. However, the two protocols are similar enough to allow for a relatively simple proxy agent to convert messages in real-time. Such a design might facilitate the speedy deployment of the core TLS and SNMPv3 security features while allowing eventual migration of the underlying controller logic to use the updated object designs (which often provide additional benefits, including better security). The proxy agent itself might be implemented as a device external to the controller or as internal functionality within the physical controller of the field device.
NTCIP_1201-289 Text

As a result, this document defines two sets of PRLs and RTMs. The first is for a native device that supports the "current" objects while the second is for a proxy device that uses objects that map directly to objects defined in previous versions of NTCIP standards.
NTCIP_1201-167 Note

While most of the deprecated objects within this document map directly to objects from previous versions, some objects have minor revisions due to changes and preciseness added in the underlying standards. For example, prior versions of NTCIP defined a few objects with a SYNTAX of INTEGER (0.. 4294967295). These objects have created problems with some SNMPv1 implementations, but this syntax is strictly prohibited in SNMPv3; as a result, within the MIBs defined in this document, they are defined with a SYNTAX of Unsigned32. This results in a different value in the "type" field of the encoding, but this is a relatively easy translation for the proxy to make when converting between the SNMPv1 and SNMPv3 formats (and can eliminate resulting ambiguities if the proxy is internal to the field device and does not need to convert to SNMPv1 format).
NTCIP_1201-373 Note

A device can support both native and proxy formats.
NTCIP_1201-169 Text

For example, the current version for exchanging data pertaining to auxiliary input/output devices is defined in the ISO 20684 general-purpose input/output (GPIO) feature. A native device supporting this capability is required to support these GPIO objects. However, a proxy is required to support the objects contained in the NTCIP1201-AuxIOv2 MIB, as defined in Section 6 for its interface to the management station. (The proxy agent can also support the NTCIP1201-AuxIO MIB, as defined in Section 7). By comparison the interface between the proxy and the SNMPv1 field device does not conform to this document at all; that interface should conform to the selected prior version of NTCIP.
NTCIP_1201-188 user need

2.2.2 Support Proxy Data Format
NTCIP_1201-189 Text

To facilitate migration to SNMPv3 while providing maximum security over the communications network, some deployments might wish to use an SNMPv3-to-SNMPv1 proxy near or in the field device controller. This would allow all communications to the proxy to be secured (using TLS for network security and SNMPv3 for data access security within the proxy agent). However, these requests would use objects defined by prior versions of NTCIP translated into a format compatible for use within SNMPv3.
NTCIP_1201-190 Text

This user need does not trace to any requirements directly, rather it is an indicator that implementations supporting this feature should complete the Proxy PRL and Proxy RTM.

When the trigger fires and the recording action is called, a user-specified fraction of the samples gathered prior to the trigger event are retained, and further samples are gathered until the recording is complete. Thus, the recording mechanism allows the management station to gather data sampled before, during, and after an event of interest. Multiple recordings may be active simultaneously and the memory available for recordings may be allocated flexibly to different classes under user control.

4.2.1 Retrieving Logged Data
NTCIP_1201-627 Text

The standardized dialog for a management station to retrieve logged data shall be as follows:
NTCIP_1201-628 List Level 1

(Precondition) The management station shall be aware of the number of events that had previously been reported for the device for the subject event class (e.g., from the previous performance of this operation).

The management station shall GET the following data:
NTCIP_1201-630 List Level 2

eventClassNumRowsInLog.x
NTCIP_1201-631 List Level 2

eventClassNumEvents.x
NTCIP_1201-632 List Level 1

If eventClassNumEvents.x has not changed since the previous reading, the management station shall exit the process. Otherwise, the management station shall determine the additional number of events that have occurred since the last read.
NTCIP_1201-633 Note

This is generally determined by subtracting the previous number of events from eventClassNumEvents; however, since this object wraps at 65535, the management station should be prepared to determine the differential if eventClassNumEvents is less than the previous number.
NTCIP_1201-634 List Level 1

The management station shall determine the lesser of eventClassNumRowsInLog and the additional number of events that have occurred since the last read. This number shall be termed the Events to Read.

NTCIP_1201-636 List Level 2

eventLogID.x.y
NTCIP_1201-637 List Level 2

eventLogTime.x.y
NTCIP_1201-638 List Level 2

eventLogValue.x.y
NTCIP_1201-639 List Level 1

Repeat the same GET operation with y decremented by one (1) for each set of duplicated values (until y reaches a value of zero (0)).
NTCIP_1201-640 Note

If the event class is full and another event occurs, the new event is recorded in the last entry and all previously logged data is moved to one index lower with index 1 being deleted from the table. Thus, if a duplicate row is detected (e.g., same event at same time), it is likely an indication that the same event is being read and that a new event was added to the log.
NTCIP_1201-641 Note

The management station may wish to clear the event log after the read to minimize the above problem.
